LogBrew stores observability data in ClickHouse. The backend is designed for
append-friendly, high-volume telemetry: logs, trace spans, issue events, action
events, status changes, and release summaries.

## Backups

Production backups use native ClickHouse `BACKUP` and `RESTORE`, then sync the
backup artifacts to Hetzner Storage Box over SSH and rsync.

```bash
logbrew-clickhouse-backup backup
logbrew-clickhouse-backup restore-check latest
logbrew-clickhouse-backup list
```

<Warning>
  LogBrew production backups do not use S3 object storage. Storage Box over
  SSH/rsync is the documented backup target.
</Warning>

## Restore checks

Backups are only useful if restores are practiced. Operators should run restore
checks regularly against a scratch database and confirm that ClickHouse can read
the restored data.
