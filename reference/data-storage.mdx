LogBrew stores observability data in ClickHouse. The backend is designed for
append-friendly, high-volume telemetry: logs, trace spans, issue events, action
events, status changes, and release summaries.

## Storage model

<CardGroup cols={2}>
  <Card title="Append first" icon="database">

    predictable.
  </Card>

  <Card title="Release-aware" icon="tag">

  </Card>

  <Card title="Grouped issues" icon="circle-alert">

  </Card>

  <Card title="Agent-readable" icon="bot">

  </Card>
</CardGroup>

## Backups

Production backups use native ClickHouse `BACKUP` and `RESTORE`, then copy the
backup artifacts to private off-site storage.

<Warning>
  Public docs describe the restore guarantee, not internal storage endpoints,
  credentials, hostnames, or provider-specific backup paths.
</Warning>

## Restore checks

Backups are only useful if restores are practiced. Operators should run restore
checks regularly against a scratch database and confirm that ClickHouse can read
the restored data.
